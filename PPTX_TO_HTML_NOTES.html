PowerPoint to HTML Converter — Notes

Q1: Reading Order (Slide 3)

Root cause: a two-pass architecture conflict with math-containing text boxes.

PowerPoint wraps any text box that contains an inline equation in an <mc:AlternateContent> element in the slide XML. python-pptx’s slide.shapes iterator silently skips these elements, so they are invisible to normal iteration.

The original converter handled shapes in two separate passes inside convert_slide:

  • Pass 1: iterates slide.shapes — processes all regular shapes (text frames, pictures, tables, groups) in XML / z-order.
  • Pass 2: scans the raw spTree XML for <mc:AlternateContent> elements and appends their content after everything from Pass 1, regardless of where they appear in document order.

On Slide 3 the “Data” text box contains inline math equations, so it is an <mc:AlternateContent> element. The graphic is a regular picture shape visible in Pass 1. Result:

Step What was rendered
Pass 1 Core constraints text → Graphic
Pass 2 Data section (with math) appended last

Even though the PPT reading order (tab order / Selection Pane) shows Data before the graphic, the converter always emitted AlternateContent shapes last. Reordering shapes in PowerPoint’s Selection Pane could not fix this.

Fix: single unified pass (implemented)

The two-pass approach has been replaced with a hybrid single-pass design in convert_slide:

  1. Build a shape_id → Shape map from slide.shapes (one cheap scan). This extracts the title, marks shapes to suppress (slide numbers, running headers), and keeps references to all python-pptx Shape objects so their full API remains available.
  2. Walk spTree direct children in document order. For each child:
    • Regular shape tags (p:sp, p:pic, p:grpSp, p:graphicFrame, p:cxnSp) — look up the corresponding Shape by ID and dispatch to the existing renderer. All image, table, text-frame, and group logic is unchanged.
    • mc:AlternateContent — render the math text box inline via render_txBody_from_xml, in its correct document position.

This preserves the full python-pptx API for all non-math shapes (no XML reimplementation of image or table rendering) while correctly interleaving math shapes with regular content. Reordering shapes in PowerPoint’s Selection Pane now has the expected effect on HTML output order.

Q2: Image Scaling

Images are sized using the shape’s actual EMU dimensions from the slide, converted to pixels at 96 DPI:

px_w = round(shape.width / 914400 * 96)
px_h = round(shape.height / 914400 * 96)
size_style = f' style="width:{px_w}px; height:{px_h}px; max-width:100%; height:auto;"'

The CSS also applies figure img { max-width: 100%; height: auto; } within a main { max-width: 900px } container.

A standard PowerPoint slide is 10 inches wide, which calculates to ~960 px at 96 DPI. A large image spanning most of the slide width will therefore be rendered at nearly the full 900 px container width — which reads as “full page” in the HTML.

--img-scale option (implemented)

An --img-scale PCT command-line option has been added. It accepts a percentage and multiplies the calculated pixel dimensions before writing the inline style. The max-width: 100% constraint is retained, so images remain responsive on narrow screens regardless of scale.

python pptx_to_accessible_html.py deck.pptx --img-scale 60
python pptx_to_accessible_html.py deck.pptx --img-scale 50 --mathjax

Default is 100 (no change from previous behavior).

Q3: Authoring Math in PowerPoint for Best OMML → MathML Conversion

The script converts OMML (Office Math Markup Language) to MathML. OMML is the internal format produced by PowerPoint’s modern equation editor. Any approach that results in OMML in the .pptx XML will convert well; anything that does not produce OMML will not convert at all.

Approach Conversion quality Notes
Insert > Equation (ribbon, point-and-click) Best Creates native OMML. The gold standard.
LaTeX input in equation editor Best Open equation editor (Alt+=), type LaTeX syntax (e.g. \frac{a}{b}), press Space to commit. PPT converts to OMML internally — identical result to point-and-click.
UnicodeMath input in equation editor Best Another shorthand within the same editor (e.g. x^2/2 + Space). Also saves as OMML.
Images of equations (screenshots, etc.) None No mathematical structure. Falls back to the <mc:Fallback> raster image in the PPTX, which the converter cannot interpret.
Legacy Equation 3.0 (Insert > Object) None OLE object embedded in the slide. Not OMML. Completely invisible to the converter.
Text characters mimicking math (x^2, /, Unicode letters) None Rendered as plain text. No semantic math structure to convert.

Practical tips for best conversion quality

  • Use a single equation object per expression. Don’t split a + b = c across multiple equation boxes or mix equation boxes with plain text runs.
  • Use semantic structures. Build fractions with the fraction operator, superscripts/subscripts with their proper operators, summations with the summation template — not workarounds using /, ^, or Unicode characters typed as plain text.
  • Subscripted and superscripted variables convert reliably. Complex multi-line arrays and matrices can be lossier depending on nesting depth.
  • Run with --mathjax for the best cross-browser MathML rendering. Native MathML support varies (especially between Safari and Firefox), and MathJax normalizes this.
  • Run with --focusable-math if keyboard and screen-reader accessibility is a priority — it adds tabindex="0" to each <math> element so users can tab directly to equations.
